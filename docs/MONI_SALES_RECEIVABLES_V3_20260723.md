# MONI 판매관리 V3 — 수금·미수금 / 매출채권

## 목적
판매 `입금상태`를 수동으로 고르는 수준에서 벗어나 실제 입금내역과 입금예정일을 기준으로 받을 돈을 관리한다.

## 기준 데이터
- 매출원장: `sales_orders`
- 실제 입금원장: `sales_receipts`
- 거래처 수금조건: `sales_clients.payment_due_*`

## 수금기일
거래처별 기본 자동 계산 규칙:
1. 자동 계산 안 함
2. 판매일 + N일
3. 익월 N일

익월 N일에서 해당 월에 없는 날짜는 말일로 조정한다.
예: 2026-01-31 판매 / 익월 31일 → 2026-02-28.

자동 규칙은 이후 신규 판매건에 적용한다. 개별 판매건은 `due_date`를 직접 수정할 수 있다.
기존 자유입력 `payment_terms`는 참고 메모로 보존하며 자동 해석하지 않는다.

## D-Day 상태
- 입금완료: 남은 미수금 0
- 연체: due_date < 오늘
- D-Day: due_date = 오늘
- D-1 ~ D-3: 3일 내 수금예정
- 예정: D-4 이상
- 예정일 미설정: due_date 없음

## 실제 입금
`payment_status` 문자열을 금액 원장으로 사용하지 않는다.
새 구조의 정확한 수금액은 `sales_receipts.status='posted'` 합계다.

입금 등록 필드:
- 입금일
- 입금액
- 방법: 계좌 / 현금 / 카드 / 기타
- 참조번호
- 비고

입금액은 현재 남은 미수금을 초과할 수 없다.

## 부분입금
- 실제 입금합계 < 판매 총액 → 일부입금
- 실제 입금합계 >= 판매 총액 → 입금완료
- 실제 입금 없음 → 미입금

API가 실제 입금원장을 기준으로 `sales_orders.payment_status`를 호환용으로 자동 동기화한다.

## 입금 취소
실제 돈 기록은 삭제하지 않는다.
입금 오류 수정은 `reversed`로 상태를 바꾸고:
- 취소시각
- 취소사유
을 보존한다.
취소된 금액은 수금합계에서 제외한다.

## 과거 호환
과거 데이터에서 실제 입금원장 없이 `paid`만 남아 있는 경우 기존 상태를 존중해 미수금 0으로 본다.
`partial`만 있고 실제 입금액이 없는 경우 금액을 추측하지 않으며 전체 판매금액을 미검증 미수금으로 표시하고 경고한다.

## Control Tower 연결
메인 대시보드는 이제 다음을 수금 API에서 가져온다.
- 현재 받을 돈
- 연체 금액/건수
- 3일 내 수금예정
- 이번 달 실제 입금
- 우선 회수 대상 리스트

## 데이터 보호
- 기존 판매/거래처 삭제 없음
- 가짜 입금 데이터 생성 없음
- 테스트는 DB transaction rollback 방식으로만 수행
- 2026-01-31 / 익월 31일 → 2026-02-28 자동 계산 검증 완료
- rollback 후 테스트 거래처/판매건 잔존 0건 확인
